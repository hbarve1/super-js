# v1.0 Workstream Agent Contract

Every `v1.0-ws-*` agent inherits these rules.

## Before you start

1. Confirm you are in the **assigned worktree** (not repo root on `main`).
2. Read `specs/mission.md` and `specs/design/git-workflow.md`.
3. Read `specs/roadmap/v1.0/manifest.json` — find your workstream entry.
4. Read your **WS-*.md** spec end-to-end; that file is your scope.
5. If an `amendment` or `agentBrief` is listed in the manifest, read it — it overrides conflicting WS text.

## While working

- **Branch:** work only on the branch named in manifest; never commit to `main`.
- **Scope:** implement acceptance criteria in the WS file; no drive-by refactors.
- **Nx:** run tasks from `superjs/` via `pnpm nx …`.
- **Conflicts:** check `manifest.json` → `conflictZones`; rebase onto `origin/main` before touching shared CI files.
- **Status:** set your workstream `status` to `in_progress` in `manifest.json` when you begin.

## When finished

1. Run every command in the WS **Acceptance criteria** section; all must pass.
2. Set `manifest.json` → `status: "pr_open"` and record the PR URL in `pr`.
3. Open a PR to **`main`**:

```bash
git push -u origin HEAD
gh pr create --base main --title "feat(v1.0): <short title>" --body "$(cat <<'EOF'
## Summary
- …

## Workstream
WS-XX — link to specs/roadmap/v1.0/WS-*.md

## Test plan
- [ ] Acceptance criteria from WS spec verified

EOF
)"
```

4. Do **not** merge your own PR unless the user asks.

## Docs single source of truth (ADR-011)

- User-facing docs live in repo-root **`docs/`** (GitHub + website).
- Website app: **`superjs/apps/website`** — do not create `superjs/apps/docs`.
- Formal spec stays in **`specs/`**.

## Out of scope

- Human-only v1.0 items (npm publish, beta program, security review, etc.) — see `specs/roadmap/v1.0/README.md`.
- Other workstreams unless explicitly blocked and you are unblocking a dependency.
