# Roadmap

SuperJS is built in three phases: a Babel-based prototype to establish the language, a custom compiler to own the full pipeline, and an LLVM native backend for production-grade performance.

## Phase 1 — Babel Prototype (Complete, v0.1.0)

**Status: Released 2026-05-30**

Phase 1 established the SuperJS language, toolchain, and type system using Babel as the compilation backbone. The goal was to ship a real, working language quickly and validate design decisions with actual code.

### What shipped

- **Unified CLI**: `superjs build`, `superjs lint`, `superjs format`, `superjs test`
- **Type checker**
  - Sound null safety: non-nullable types by default, `T?` for nullable (SJS-E001)
  - Type mismatch detection on assignment and return (SJS-E002)
  - Implicit-any warnings in strict mode (SJS-W001, `--strict`)
- **Preprocessor**
  - Sum type declaration syntax (`type Result<T, E> = Ok(T) | Err(E)`) compiled to discriminated unions
  - Match expression syntax compiled to exhaustive switch IIFEs
  - Exhaustiveness checking for match over sum types (SJS-E007)
- **Build flags**: `--watch`, `--strict`, `--no-emit`, `--sourcemap`, `--json`, `--target`
- **Config file**: `superjs.config.json`
- **JSX**: enabled by default, configurable factory and fragment
- **Demo projects**: algorithms (data structures), web (todo app), JSX/React (markdown editor), Node.js (code stats analyzer)
- **202 tests passing across 18 suites**

## Phase 2 — Custom JS Compiler (Next, v1.0)

**Status: Planned**

Phase 2 removes the dependency on Babel and TypeScript's parser. Every part of the compilation pipeline will be written from scratch in SuperJS/TypeScript, giving full control over syntax, error messages, and performance.

### Planned work

- **Hand-written lexer** — zero external dependencies
- **Recursive-descent parser** — full SJS grammar, not TypeScript's grammar
- **Full SJS syntax enforcement** — stricter rejection of banned features (`any`, mapped types, conditional types, intersection types, `enum`, `namespace`)
- **Cross-file type inference** — understand imports and module boundaries
- **Better error messages** — source spans, did-you-mean suggestions, contextual hints
- **Watch mode improvements** — incremental per-file recompilation
- **LSP server** — Language Server Protocol implementation for editor integration
- **VS Code extension** — syntax highlighting, inline diagnostics, hover types, go-to-definition

---

## Phase 3 — LLVM Native Backend (Future, v2.0)

**Status: Planned**

Phase 3 is an optional native compilation path for applications that need maximum performance. The SJS frontend from Phase 2 feeds into a C++17 compiler that targets LLVM 17.

### Planned work

- **C++17 compiler frontend** consuming Phase 2's AST
- **LLVM 17 IR code generator**
- **Monomorphized generics** — generic types specialized at compile time, no boxing
- **Native binary output** — standalone executables, no Node.js required
- **WebAssembly target** — compile SJS to `.wasm`
- **Compile speed target** — under 50ms for large projects
