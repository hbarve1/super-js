# SuperJS Benchmark Suite

This directory contains the benchmark infrastructure for the SuperJS compiler pipeline.
Benchmarks are run with [tinybench](https://github.com/tinylibs/tinybench) and results
are committed to `bench/baseline.json` after each sprint's done-signal.

---

## What is benchmarked

The benchmark suite covers the five compiler phases that make up a cold compile:

| Phase | Metric | Target |
|-------|--------|--------|
| **Lexer** | tokens/sec | ≥ 500 000 |
| **Parser** | AST-nodes/sec | ≥ 200 000 |
| **Type-checker** | constraints/sec | ≥ 50 000 |
| **Codegen** | AST-nodes/sec | ≥ 100 000 |
| **Cold compile (10 k LOC)** | total ms | ≤ 2 000 |

Per-phase targets and measurement methodology are documented in
[`bench/per-phase-targets.md`](./per-phase-targets.md).

---

## How to run

From the **repository root**:

```bash
npm run bench
```

This runs the full tinybench suite and prints results to stdout. To record a new
baseline, pipe the output to `bench/baseline.json`:

```bash
npm run bench -- --json > bench/baseline.json
```

> **Note:** Always run benchmarks on a quiet machine (no other heavy processes).
> The CI regression check compares against `baseline.json` using the same Node.js
> version and platform, so baseline recordings on your laptop may differ from CI
> numbers. Use the CI baseline for hard regression gating; use local runs for
> development iteration.

Individual phase benchmarks can be run in isolation:

```bash
# Lexer only
npm run bench -- --filter lexer

# Parser only
npm run bench -- --filter parser

# Type-checker only
npm run bench -- --filter type-checker

# Codegen only
npm run bench -- --filter codegen
```

## `baseline.json` format

`bench/baseline.json` stores the tinybench output in a documented schema:

```jsonc
{
  "_comment": "Placeholder — replace with output of `npm run bench` on first run",
  "recorded": "YYYY-MM-DD",          // ISO date of the baseline run
  "compiler_version": "0.x.x",       // superjs package version
  "platform": "darwin-arm64",        // os.platform() + os.arch()
  "benchmarks": [
    {
      "name": "lexer",               // phase name (matches --filter key)
      "hz": 512345.6,                // operations per second (tinybench .hz)
      "mean": 0.00000195,            // mean time per operation in seconds
      "unit": "tokens/sec",          // human label for the metric
      "target": 500000               // minimum acceptable hz value
    }
    // ... one entry per phase
  ]
}
```

The `hz` field is the primary regression signal. CI computes:

```
regression_pct = (baseline.hz - current.hz) / baseline.hz * 100
```

and applies the thresholds defined in [`bench/per-phase-targets.md`](./per-phase-targets.md).

## How CI uses benchmarks

The CI benchmark step runs on every sprint's done-signal PR (not just Sprint 1.6):

1. **Run benchmark** against the current code.
2. **Compare** each phase's `hz` against the corresponding entry in `bench/baseline.json`.
3. **Soft-fail (+10 % regression):** CI step emits a warning annotation on the PR. The PR can still be merged, but the annotation is visible in the PR check summary.
4. **Hard-fail (+25 % regression):** CI step fails with exit code 1. The PR cannot be merged until the regression is investigated and resolved (or the baseline is deliberately updated with a justification comment in the PR).

The hard determinism gate at Sprint 1.6 additionally verifies that two consecutive
benchmark runs of the same fixture produce byte-identical compiler output.

### Updating the baseline

When a sprint intentionally changes performance characteristics (e.g. a new optimisation
makes the parser faster, or a new phase is added), update `bench/baseline.json` as
part of the sprint's done-signal PR:

```bash
npm run bench -- --json > bench/baseline.json
git add bench/baseline.json
git commit -m "bench: update baseline after Sprint X.Y optimisation"
```

The PR description must include a brief explanation of why the baseline changed.

## Fixture used

Benchmarks run against `tests/fixtures/bench/10k-loc.sjs` — a 10 000-line SJS file
generated by `scripts/gen-bench-fixture.ts`. The fixture covers:

- All primitive types and type annotations
- Function declarations with generic type parameters
- Sum type declarations and `match` expressions
- Structural interface definitions
- Import/export statements
- JSX expressions (React-style)
- Async/await patterns

The fixture is deterministic (seeded random generation) so benchmark results are
comparable across machines and over time.
