# SuperJS v2.0 — Native Compiler Vision

**Status:** Pre-planning (post-v1.0)  
**Depends on:** v1.0 GA + self-hosting milestone  
**Theme:** "Write once, run anywhere — JS, native binary, or WASM"

---

## The One-Paragraph Vision

SuperJS v2.0 ships a **self-hosted, multi-target compiler**: written in SJS itself, producing ES2022 JavaScript (Node/Web/Workers), native OS binaries (via LLVM), or WebAssembly modules (WASM/WASI) from the same source file. A developer writes one `.sjs` file. `superjs build` produces a JS bundle. `superjs build --target native` produces a single statically-linked binary — no runtime, no VM, no install step — similar to `go build` or `cargo build`. `superjs build --target wasm` produces a `.wasm` module runnable in browsers, Node.js, or any WASI-compatible runtime. The language stays the same. Only the backend changes.

---

## Why v2.0 and Not v1.0

v1.0 is a compile-to-JS language. Its compiler is a JavaScript program that outputs JavaScript. That is the right first step — it ships fast, leverages the existing Node.js ecosystem, and lets us validate the language design against real users before committing to a native backend.

v2.0 makes SJS a **first-class systems-capable language**:
- Native binaries for CLIs, servers, and tools with no Node.js dependency
- WASM modules for browser-native compute, plugin sandboxes, and edge runtimes (Fastly, WASI)
- Performance ceiling lifts: LLVM can produce code 5–20× faster than V8 for CPU-bound work
- Single distributable binary — `curl | sh` deployments, Docker scratch images, serverless cold-start reduction
- WASM distributable — embed SJS logic in any language that has a WASM host (Python, Ruby, C, Swift)
- Self-hosting validates the language is expressive enough to build a compiler in it

## Self-Hosting (Gate to v2.0)

Before the LLVM backend, the compiler rewrites itself in SJS. The prototype and Stage 1 compiler are written in JavaScript. The v2.0 gate requires:

1. `packages/compiler-sjs/` — the compiler, rewritten in SJS
2. **Bootstrap check:** Stage-1 JS compiler compiles `compiler-sjs/` → produces `compiler-v2`; `compiler-v2` compiles `compiler-sjs/` → byte-identical output. This is the bootstrapping proof.
3. `compiler-v2` replaces `packages/compiler/` as the default `superjs build` backend
4. Prototype (`backends/prototype/`) deleted

This mirrors Go's bootstrap story (Go 1.5 rewrote the compiler in Go itself). The SJS compiler is the first large real-world SJS program — any language gap found here shapes the language before v2.0 GA.

## Multi-Target Architecture

```
.sjs source
  → Lexer (shared)
  → Parser (shared)
  → Type Checker (shared)
  → Desugaring / IR lowering (SJS-IR)
       ├── JS Backend      → ES2022 + source maps        (v1.0, unchanged)
       ├── LLVM Backend    → LLVM IR → native binary      (v2.0)
       └── WASM Backend    → WASM binary (.wasm + WASI)   (v2.0)
```

The frontend (lexer → parser → checker → IR) is shared across all backends. SJS-IR is the extension point — backends are pluggable. Both LLVM and WASM backends are first-class targets in v2.0.

### Intermediate Representation (SJS-IR)

SJS-IR is a typed, expression-oriented IR inspired by LLVM IR and MIR (Rust's mid-level IR):

- All types resolved (no inference at IR level)
- Sum types lowered to tagged structs
- Nullable types lowered to `Option<T>` tagged structs or raw pointer + null bit
- Closures lowered to structs + function pointers
- Match expressions lowered to switch + exhaustiveness proof retained as metadata
- Generics monomorphized (one concrete instantiation per unique type argument)

SJS-IR is human-readable, serializable to `.sjsir` files, and versionable. This enables:
- Separate compilation (compile IR of a module, link later)
- Cross-language LTO in theory (LLVM IR merges)
- IR-level tooling (optimizers, analyzers) as a separate project

## LLVM Backend

**Target:** LLVM 18+ via `llvm-sys` bindings (or direct C FFI to LLVM-C API from SJS).

### What LLVM gives us
- Native binaries for macOS (arm64/x86_64), Linux (x86_64/arm64), Windows (x86_64)
- Cross-compilation via LLVM target triples (same as `clang --target`)
- LLVM optimization passes (O0/O1/O2/O3/Os/Oz — same flags as clang)
- Debug info (DWARF) for native debuggers (lldb, gdb)
- Link-time optimization (LTO) across modules
- Address sanitizer, thread sanitizer, undefined behavior sanitizer

### Compilation model

```bash
# JS output (v1.0 behaviour, unchanged)
superjs build src/main.sjs --outDir dist/

# Native binary
superjs build src/main.sjs --target native
# → dist/main  (statically linked, no runtime required)

# Cross-compile
superjs build src/main.sjs --target native --arch aarch64 --os linux
# → dist/main-linux-arm64

# Shared library (.so / .dylib / .dll)
superjs build src/lib.sjs --target native --emit shared
```

### Runtime for native targets

Native binaries require a minimal runtime (`@superjs/native-runtime`):
- Stack unwinding for `panic` (uses platform unwind tables)
- `Result`/`Option` variant representation (tagged union structs in C layout)
- `inspect` formatting for REPL / debug builds
- Allocator hooks (default: system `malloc`; pluggable via `#[allocator]` attribute — post-v2.0)
- I/O: `print`, `eprint`, `stdin` wired to libc `write`/`read`

No garbage collector. Memory model follows value semantics for primitives and explicit ownership for heap values — similar to Go's escape analysis. Closures capturing heap values are reference-counted (initial implementation; ownership types post-v2.0).

### What does NOT work on native targets (v2.0)

| Feature | Native support | Notes |
|---------|---------------|-------|
| JSX | No | DOM is a browser API; no native target for it |
| `dynamic` | Partial | Allowed but no V8 engine — `dynamic` values use a boxed type system at runtime |
| npm packages | No | npm packages are JS; native targets can only use `.sjs` code and `extern` C bindings |
| `async`/`await` | Yes (via green threads) | Backed by a cooperative runtime, not libuv |
| Closures | Yes | Reference-counted captures |
| Generics | Yes | Monomorphized |
| Sum types | Yes | Tagged unions in C layout |
| `match` | Yes | Compiled to LLVM switch |
| Interfaces | Yes | Static dispatch (vtable in v2.1 for dynamic dispatch) |

## WASM Backend

**Target:** `wasm32-wasi` (WASI Preview 2) and `wasm32-unknown` (browser, no WASI).

WASM is generated directly from SJS-IR — not via LLVM. Direct WASM codegen produces smaller, faster-to-compile output than going through LLVM's WASM backend. LLVM's WASM target remains available via `--target wasm-llvm` for cases where LLVM optimizations are worth the compile-time cost.

### What WASM gives us
- Run SJS code in browsers with no JavaScript bridge (call SJS functions from JS via the WASM module API)
- Edge runtime deployment: Fastly Compute, WASI-capable runtimes, WasmEdge, wasmtime
- Plugin sandboxes: embed SJS logic as an untrusted plugin in a host application
- Universal distributable: any language with a WASM host (Python, Ruby, C, Go, Swift) can call SJS code
- Deterministic execution: no OS-specific behavior; reproducible byte-for-byte output

### Compilation model

```bash
# WASM module (WASI — runs in wasmtime, WasmEdge, Fastly, Node.js with WASI)
superjs build src/main.sjs --target wasm
# → dist/main.wasm  (WASI Preview 2)

# Browser WASM (no WASI, no filesystem/network access)
superjs build src/lib.sjs --target wasm --wasm-env browser
# → dist/lib.wasm + dist/lib.wasm.d.ts  (typed JS bindings auto-generated)

# WASM component (WebAssembly Component Model — composable modules)
superjs build src/lib.sjs --target wasm --wasm-env component
# → dist/lib.component.wasm

# Optimize for size (uses wasm-opt via Binaryen)
superjs build src/lib.sjs --target wasm --opt size
# → dist/lib.wasm  (Binaryen Os pass applied)
```

### TypeScript bindings auto-generated

When compiling `--target wasm --wasm-env browser`, SJS emits a `.d.ts` + thin JS loader alongside the `.wasm`:

```ts
// dist/lib.wasm.d.ts  (auto-generated)
export declare function add(a: number, b: number): number;
export declare function greet(name: string): string;
```

```js
// usage from JS
import { add, greet } from "./dist/lib.wasm"
console.log(add(1, 2))      // 3
console.log(greet("world")) // "Hello, world!"
```

SJS types map to WASM types via the Component Model interface type system — `string`, `number`, `boolean`, `Result<T,E>`, `Option<T>`, and sum types all have canonical Component Model representations.

### WASI standard library

WASM targets on WASI use `@superjs/std-wasi` instead of `@superjs/std-native`:

```sjs
import { File, Stdout } from "@superjs/std-wasi/io"
import { Env, Args } from "@superjs/std-wasi/sys"

const name = Args.get(1) ?? "World"
Stdout.writeLine(`Hello, ${name}!`)
```

`@superjs/std-wasi` maps directly to WASI Preview 2 host functions. No libc. No OS threads. Green-thread concurrency backed by the WASM stack-switching proposal (where available; poll-based fallback).

### What does NOT work on WASM targets

| Feature | WASM support | Notes |
|---------|-------------|-------|
| JSX | No | Browser WASM can call JS DOM; direct DOM manipulation via `extern` bindings only |
| `dynamic` | No | No V8; `dynamic` banned at compile time for WASM targets (use sum types) |
| npm packages | No | WASM targets can only use `.sjs` code and WASI host functions |
| `async`/`await` | Yes (poll-based) | Full green threads available where WASM stack-switching is supported |
| Closures | Yes | Value-captured; no RC needed in linear-memory model |
| Generics | Yes | Monomorphized |
| Sum types | Yes | Component Model canonical form |
| `match` | Yes | Compiled to WASM br_table |
| Interfaces | Yes | Static dispatch; Component Model interface types for cross-language calls |
| C FFI (`extern`) | Yes | WASI imports; Component Model imports for non-C hosts |

### WASM module size targets

| Scenario | Target size |
|----------|-------------|
| "Hello world" | < 50 KB (after `wasm-opt -Os`) |
| Simple HTTP handler (Fastly) | < 500 KB |
| Full stdlib included | < 2 MB |

## Native Standard Library

npm packages don't exist on native targets. v2.0 ships `@superjs/std-native` — a native I/O and OS layer:

### `@superjs/std-native/io`
```sjs
import { File, Stdin, Stdout, Stderr } from "@superjs/std-native/io"

const content: Result<string, Error> = File.readText("input.txt")
Stdout.writeLine("hello from native SJS")
```

### `@superjs/std-native/net`
```sjs
import { TcpListener, TcpStream } from "@superjs/std-native/net"

const listener = TcpListener.bind("0.0.0.0:8080").unwrap()
for const conn in listener.accept() {
  handleConn(conn)
}
```

### `@superjs/std-native/os`
```sjs
import { Env, Args, Process } from "@superjs/std-native/os"

const port = Env.get("PORT") ?? "3000"
const args = Args.all()
Process.exit(0)
```

### `@superjs/std-native/fs`
```sjs
import { Dir, Path } from "@superjs/std-native/fs"

for const entry in Dir.read(".") {
  const path = entry.path
  const size = entry.metadata().size
}
```

### `@superjs/std-native/sync`
```sjs
import { Mutex, Arc } from "@superjs/std-native/sync"

// Shared mutable state across green threads
const counter = Arc(Mutex(0))
```

## Interop with C / System Libraries

Native SJS can call C functions via `extern`:

```sjs
extern "C" {
  fn malloc(size: number): dynamic  // raw pointer as dynamic
  fn free(ptr: dynamic): void
  fn strlen(s: string): number
}

const len = strlen("hello")  // calls libc strlen directly
```

`extern` blocks are unsafe by design — wrapped in a `@unsafe` annotation that lint rule L001 flags in normal code. The type system does not protect across the FFI boundary (same as Rust's `unsafe`).

## Distribution

### Single static binary
`superjs build --target native` produces a statically linked binary by default (like Go). No shared library dependencies beyond libc. Binary size: target < 5 MB for a simple HTTP server, < 15 MB for a full compiler invocation.

### Cross-compilation matrix
v2.0 supports cross-compilation from any host to any target:

| Target | Host | Status |
|--------|------|--------|
| `x86_64-linux-musl` | Any | Tier 1 |
| `aarch64-linux-musl` | Any | Tier 1 |
| `x86_64-macos` | macOS | Tier 1 |
| `aarch64-macos` | macOS (M-series) | Tier 1 |
| `wasm32-wasi` (WASI Preview 2) | Any | Tier 1 |
| `wasm32-unknown` (browser) | Any | Tier 1 |
| `wasm32-wasi-component` | Any | Tier 1 |
| `x86_64-windows` | Any | Tier 2 |
| `wasm64-wasi` | Any | Tier 2 (post-v2.0) |

Cross-compilation uses LLVM target triples + musl libc (for Linux static builds). `zig cc` as the linker for cross-compilation is evaluated as an alternative to maintain a clang install.

### `superjs install` — like `go install`

```bash
superjs install github.com/hbarve1/my-tool@latest
# → compiles from source, installs to ~/.superjs/bin/my-tool
```

Works for any public `.sjs` project with a `superjs.config.json`. No pre-built binary hosting required — source is the distribution unit.

### Docker scratch images

```dockerfile
FROM scratch
COPY --from=build /app/dist/server /server
ENTRYPOINT ["/server"]
```

Static SJS binaries run in `scratch` containers. Final image: < 20 MB for a typical HTTP server.

## Development Experience for Native Targets

### Build speed
- Incremental by default (same `apiHash`/`docHash` system as v1.0)
- Cold build of 10k LOC ≤ 10 s (LLVM is slow; O0 debug builds ≤ 3 s)
- Parallel LLVM module compilation (one LLVM module per `.sjs` file)

### Debug builds
- DWARF debug info included by default
- `lldb` and `gdb` work out of the box
- Source maps from `.sjs` to LLVM IR (intermediate); `.sjs` to binary line numbers

### Native test runner
`superjs test --target native` compiles and runs tests natively. No Jest, no Node.js. Tests use the same `@superjs/test` framework surface but run as native binaries.

### Hot reload (development)
Not available for native targets — binaries must be relinked. Development workflow: use `--target js` for rapid iteration; `--target native` for production builds and benchmarks.

## v2.0 Success Criteria

- SJS compiler compiles itself (bootstrap proof passes)
- `superjs build --target native` produces a running binary on macOS arm64 + Linux x86_64
- `superjs build --target wasm` produces a `.wasm` module that runs in wasmtime and in a browser
- A "hello world" HTTP server in SJS compiles to a static binary < 5 MB that passes load testing

- Cross-compilation from macOS to Linux x86_64 works out of the box
- `superjs install` works for public `.sjs` projects
- Build time for 10k LOC native target ≤ 10 s at O0, ≤ 30 s at O2
- Build time for 10k LOC WASM target ≤ 5 s (direct IR codegen)
- Docker scratch image demo published
- Fastly Compute demo: SJS WASM handler deployed to Fastly edge

## Open Questions (to resolve before v2.0 planning starts)

1. **Reference counting vs tracing GC for native?** RC is simpler to implement; tracing GC (like Go's) is faster for graph-heavy data. Decision deferred until self-hosting is working and real memory patterns are measurable.
2. **LLVM-C API or LLVM IR text?** Binding to LLVM-C from SJS is possible via `extern`; emitting LLVM IR text and shelling out to `llc` is simpler to bootstrap. Likely: text IR first, C API later.
3. **Green thread runtime vs OS threads?** Green threads (like goroutines) scale better and avoid mutex overhead; OS threads are simpler to implement. v2.0 uses green threads backed by a work-stealing scheduler.
4. **`zig cc` as cross-compilation linker?** Zig ships a hermetic `clang` + musl; using it would eliminate the need for users to install a cross-compilation toolchain. Evaluate against licensing constraints.
5. **Module system for native targets?** npm modules don't exist. Options: (a) SJS-native package registry; (b) git-based installs (like Go modules); (c) SJS-flavored `pkg.sjs.dev` registry. Likely: git-based v2.0, registry post-v2.0.
6. **Direct WASM IR codegen vs LLVM WASM backend?** Direct codegen produces smaller output and faster compile times; LLVM WASM backend enables advanced optimizations (LICM, vectorization). Plan: direct IR for release builds with `-Os`/`-Oz`; LLVM WASM backend available via `--target wasm-llvm` for throughput-critical modules.
7. **WASM GC proposal vs linear memory?** WASM GC (now stage 4, shipping in all major browsers) allows SJS sum types and closures to map directly to reference types — no manual RC in WASM. Evaluate for browser `--wasm-env browser` target; linear memory stays for WASI/server targets where GC is not available.
8. **`dynamic` in WASM?** Currently banned. Could be supported via a boxed-value representation using `anyref` (WASM GC). Deferred — `dynamic` in WASM adds significant complexity and is likely rare given WASM's use cases.
